GUI Agent RL 论文地图

这篇综述的核心观点很明确：GUI agent 不再只是 SFT 后做推理，而是在向闭环 RL 训练迁移。原因有三类：第一，GUI 任务天然是长时序、稀疏终止奖励、局部可观测；第二，真实界面会变化，静态示范会产生 covariate shift；第三，GUI 成功与否常能用环境状态、DOM、数据库、文件状态或坐标命中来验证，因此很适合 RLVR。

Taxonomy

三条主线

1. Offline RFT / Offline RL用静态轨迹、偏好对或可验证标签做 DPO/GRPO/IQL/CQL，解决安全和成本问题。代表是 Digi-Q、UI-R1、GUI-R1、Agent Q。
2. Online RL在真实或半真实环境里采样，用课程学习、ORM/VLM judge、GRPO/PPO/AWR 等做多轮优化。代表是 DigiRL、WebRL、UI-TARS/ARPO、UI-TARS-2、MobileRL、WebAgent-R1、ZeroGUI、DART-GUI、ComputerRL。
3. Hybrid / Model-Based / Self-Evolution先用离线数据冷启动，再用在线交互、世界模型、层级规划、API-GUI 混合动作或自迭代奖励把能力放大。代表是 UI-S1、Hi-Agent、DynaWeb、UltraCUA、Co-EPG、WebWorld、Code2World、ClawGUI。

Environments

环境、容器与框架

2.1 Web / Browser 环境

常用环境：

  • MiniWoB++：早期 HTML5 synthetic micro-task sandbox，奖励稀疏但环境可并行，适合验证基础 RL。
  • WebShop：模拟电商，奖励可由商品属性重叠给出，适合语言 agent 与商品搜索。
  • Mind2Web / Multimodal-Mind2Web：真实网页快照与 DOM 标注，主要用于离线训练和 grounding/action candidate compression。
  • WebArena / VisualWebArena：自托管“模拟互联网”，包含购物、论坛、GitLab、地图等服务，能用后端脚本验证数据库/页面状态。
  • BrowserGym：把 WebArena、MiniWoB++、VisualWebArena、WebChoreArena 等统一成 gym-like API，提供 observation/action 标准化。

容器与框架：

  • WebArena/VisualWebArena 一般需要自托管服务，常见部署是 Docker/Docker Compose 起多个 Web service，再由 Playwright/浏览器自动化驱动页面。
  • BrowserGym/AgentLab 是 Python + Gymnasium 风格接口，底层常用 Playwright/Chromium，observation 包括目标、URL、DOM、accessibility tree、截图。
  • RL 框架侧常把 browser env 当 rollout worker，训练端用 GRPO/PPO/AWR/DPO 等；大规模时用异步队列隔离环境 I/O 和 GPU learner。

2.2 Desktop / OS 环境

常用环境：

  • OSWorld：真实 Ubuntu/桌面应用、多 App、多文件状态任务，369 个开放式桌面任务。
  • ScreenAgent / WindowsAgentArena / OfficeWorld：更偏桌面纯视觉、Windows 或 Office 场景。
  • TheAgentCompany、OmniACT、OSWorld-MCP、MCPWorld：企业任务、桌面+网页、API/GUI 混合评测。

容器与框架：

  • OSWorld 类环境常用 Docker + QEMU/虚拟机 + virtual framebuffer/VNC 截屏。Docker 负责隔离任务和依赖，QEMU/VM 提供真实 OS 环境，VNC/截图通道提供视觉 observation。
  • 控制动作常通过 PyAutoGUI、xdotool、VNC pointer/keyboard、Accessibility API 或系统 API。ComputerRL/UltraCUA/MAI-UI 开始把 GUI 原子动作和高层 API/MCP tool 合并成 hybrid action space。
  • 成功奖励不只看文本输出，而是检查文件、数据库、浏览器状态、应用侧 effect。

2.3 Mobile / Android 环境

常用环境：

  • AndroidEnv：标准 RL MDP for Android，连续触摸/手势，适合经典 RL，但时序长。
  • AndroidWorld：真实 Android App、116 个 programmatic tasks、动态参数化任务，用 ADB/SQLite/文件状态检查 reward。
  • AndroidLab、Mobile-Env、UISim、GUI Exploration Lab、MobileWorld：覆盖移动真实 App、模拟 UI 或导航图。

容器与框架：

  • 主体是 Android Emulator / AVD + ADB，而不是传统 Docker 容器。Docker 可以包外层服务，但真正的环境隔离来自 emulator snapshot、设备镜像、任务 setup/teardown。
  • 奖励常用 ADB 查询、SQLite 数据库检查、文件系统检查、Accessibility tree 或后台 evaluator。
  • 并行训练会启动几十到几百个 emulator，CPU/内存/I/O 是主要瓶颈；GPU 主要给 VLM inference/training。

2.4 世界模型 / Latent Rollout

代表：DynaWeb、WebSynthesis、WebWorld、Code2World、SimURA、UI simulator。

技术方案：

  • 学一个 action-conditioned dynamics model：输入当前 UI 表示和动作，预测下一网页/屏幕/DOM/可渲染代码。
  • 用 world model 做 MCTS、dream rollout 或 synthetic trajectory generation，再周期性回到真实环境校准。
  • Code2World 走 renderable code 路线，用 HTML/代码作为可控状态表示；WebWorld/DynaWeb 走网页状态生成路线；WebSynthesis 结合世界模型和 MCTS。

价值：

  • 绕过 live GUI 的 I/O wall。综述估算真实 web/desktop/mobile 单环境通常只有 0.2-2 steps/s，而 latent/world-model rollout 可到 10-100+ steps/s，仿真并行可达 10^3-10^4 steps/s 量级。
  • 主要风险是 sim-to-real gap，所以有效 pipeline 通常是：离线数据预训练 -> 世界模型/模拟环境训练 -> 少量真实在线校准。

2.5 RL 算法与训练系统

主流算法：

  • GRPO/RLVR：当前 GUI RL 最常见。优势是不用 critic，适合长视觉序列和大 VLM；代表 GUI-R1、UI-R1、InfiGUI-G1、GUI-Eyes、Mano。
  • DPO/Preference：适合从成功/失败轨迹对学习，训练稳定；代表 Agent Q、UI-TARS 部分阶段。
  • AWR/IQL/CQL/Q-learning：适合离线轨迹和 value reranking；代表 DigiRL、Digi-Q。
  • MCTS/Tree Search：适合轨迹合成和高质量数据搜索；代表 Agent Q、WebSynthesis。
  • Process/Progress Reward：解决长任务稀疏奖励；代表 ProgRM、Step-GUI、ClawGUI PRM、AgentPRM。

训练系统：

  • veRL、OpenRLHF、ReaL、AReaL、HybridFlow 是常见 RLHF/RLVR 系统底座。
  • vLLM/PagedAttention 用于 rollout inference；训练端常用 PyTorch FSDP、Megatron-LM 或 DeepSpeed；模型权重在 rollout actor 和 trainer 之间异步同步。
  • DART-GUI、ComputerRL、MAI-UI、AReaL 类系统都采用 actor-trainer 分离：CPU/浏览器/模拟器 worker 连续采样，GPU learner 从 replay buffer 消费轨迹。
  • 真正的扩展瓶颈是环境 I/O：截图、渲染、网络、DOM/状态解析、ADB/VM 文件 I/O，而不是单纯 GPU 数量。

Recipe

现在大家实际怎么做

一个比较标准的 pipeline 是：

  1. 数据冷启动：用 AitW、AndroidControl、Mind2Web、OmniACT、ScreenSpot、Ferret-UI、Explorer/AgentTrek/OS-Genesis 合成轨迹做 SFT 或 offline RFT。
  2. 统一动作空间：把 action 规范化为 click(x,y)、type(text)、scroll、swipe、back/home、finish，或进一步加入 API/MCP/tool calls。
  3. 奖励设计：低层 grounding 用坐标命中/IoU/高斯连续奖励；中层过程用 progress/process reward；终局用数据库、文件、URL、DOM、ADB 状态或 VLM-as-judge。
  4. 在线采样：Web 用 BrowserGym/WebArena + Playwright + Docker 服务；Desktop 用 OSWorld + Docker/QEMU/VNC；Mobile 用 Android emulator + ADB。
  5. 训练算法：小到中模型多用 GRPO/RLVR；需要偏好稳定时用 DPO；有离线轨迹时训练 value model 或用 AWR；长任务用 curriculum、replay、subgoal、PRM。
  6. 系统扩展：把环境 rollout 和模型训练异步解耦；大量环境 worker 并行；vLLM 做快速采样，FSDP/Megatron 做训练，队列或 replay buffer 管理轨迹。
  7. 真实部署：走 hybrid API-GUI，能调用 API 就调用 API，GUI 作为通用 fallback；加 verifier/backtracking/memory，避免一次错误导致整条任务失败。

Trends

关键趋势

  • 从“模仿人类点击”转向“可验证环境反馈上的自我改进”：GUI 比普通文本生成更适合 RLVR。
  • 从二值奖励转向连续、过程、多层奖励：GUI-G2/InfiGUI-G1/UI-AGILE/ProgRM/Step-GUI 都在解决 sparse reward。
  • 从单步 grounding 转向多轮任务和恢复：WebAgent-R1、MobileRL、DART-GUI、BacktrackAgent、Hi-Agent 是这条线。
  • 从 live rollout 转向 world model 和半在线：DynaWeb、WebWorld、Code2World、UI-S1 都是在绕 I/O wall。
  • 从纯 GUI 转向 API-GUI/MCP 混合：UltraCUA、ComputerRL、MAI-UI、ClawGUI 说明真实部署不会只靠鼠标键盘。
  • 从大模型 brute force 转向小模型高效 RL：AgentCPM-Explore、GUI-R1-3K、GUI-Eyes-3K 展示了少数据和小模型的空间。
