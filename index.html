GUI Agent RL 论文地图

这篇综述的核心观点很明确：GUI agent 不再只是 SFT 后做推理，而是在向闭环 RL 训练迁移。原因有三类：第一，GUI 任务天然是长时序、稀疏终止奖励、局部可观测；第二，真实界面会变化，静态示范会产生 covariate shift；第三，GUI 成功与否常能用环境状态、DOM、数据库、文件状态或坐标命中来验证，因此很适合 RLVR。

Taxonomy

三条主线

1. Offline RFT / Offline RL用静态轨迹、偏好对或可验证标签做 DPO/GRPO/IQL/CQL，解决安全和成本问题。代表是 Digi-Q、UI-R1、GUI-R1、Agent Q、ARPO/UI-TARS。
2. Online RL在真实或半真实环境里采样，用课程学习、ORM/VLM judge、GRPO/AWR 等做多轮优化。代表是 DigiRL、WebRL、MobileRL、WebAgent-R1、ZeroGUI、DART-GUI、ComputerRL。
3. Hybrid / Model-Based / Self-Evolution先用离线数据冷启动，再用在线交互、世界模型、层级规划、API-GUI 混合动作或自迭代奖励把能力放大。代表是 UI-S1、Hi-Agent、DynaWeb、UltraCUA、Co-EPG、WebWorld、Code2World、ClawGUI。

DigiRL: Training In-The-Wild Device-Control Agents with Autonomous Reinforcement Learning

Online RLMobile
论文链接

动机

移动设备控制的数据很难只靠静态人类示范解决，因为真实 App 有随机性、状态漂移、异步反馈和任务参数变化。纯 SFT 在 Android-in-the-Wild 这类任务上会受限。

方案

DigiRL 是一个典型 offline-to-online RL 框架。先在 AitW 轨迹上用离线 RL/行为先验初始化，再搭建可并行 Android 学习环境，用 VLM-based evaluator 判断任务完成，结合 advantage-weighted RL、考虑随机性的 advantage estimator 和自动 curriculum 做在线微调。

实验

在 AitW 上，1.3B VLM 从 SFT 的 17.7% 提升到 67.2% 成功率，超过 AppAgent+GPT-4V 的 8.3%、CogAgent-17B 的 38.5% 和过滤 BC 的 57.8%。它是后续很多 GUI RL 工作的基线范式：离线冷启动 + 在线反馈。

Digi-Q: Learning Q-Value Functions for Training Device-Control Agents

Offline RLMobile
论文链接

动机

真实移动端在线交互成本高，不适合频繁 rollout；但只做 BC 又无法判断哪个候选动作长期收益更好。

方案

Digi-Q 在冻结的 VLM 中间层特征上训练轻量 Q-value head，用 offline TD learning 学动作价值。先做一次 representation fine-tuning，让 VLM 特征覆盖可操作信息；推理时从当前策略采样多个动作，用 Q 函数 Best-of-N 重排，等价于用推理时计算替代在线环境交互。

实验

在 Android-in-the-Wild 用户级设备控制任务上比此前最佳离线方法提升 21.2%，部分场景接近需要真实交互的 RL 方法。它说明 offline value model 对 GUI agent 很有价值，尤其适合不允许真实试错的场景。

Agent Q: Advanced Reasoning and Learning for Autonomous AI Agents

Offline RLWeb
论文链接

动机

Web agent 需要多步决策，单靠专家示范 SFT 容易 compounding error，且探索数据有限。

方案

Agent Q 结合 guided MCTS、自我 critique 和 off-policy DPO。MCTS 在网页交互状态树中搜索，用 value model 剪枝；成功路径作为偏好正例，失败路径作为负例，让策略从好坏轨迹对中学习。

在 WebShop 中超过 BC 和 RFT baseline；在真实网页预订场景中，Llama-3-70B 零样本成功率从 18.6% 提升到 81.7%，加入在线搜索后到 95.4%。它代表“search is data”：用搜索合成高质量偏好轨迹。

WebRL: Training LLM Web Agents via Self-Evolving Online Curriculum RL

Online RLWeb
论文链接

动机

开放 LLM 作为 Web agent 时，决策能力弱、任务少、反馈稀疏，且在线学习中策略容易漂移。

方案

WebRL 做 self-evolving online curriculum：从失败样本生成更适合当前能力区间的新任务；训练 Outcome-Supervised Reward Model 判断最终状态；用 adaptive RL 和 replay/filtering 控制策略漂移。

UI-TARS / ARPO: End-to-End Policy Optimization for GUI Agents with Experience Replay

Offline RL
论文链接论文链接

动机

跨平台 GUI agent 任务是长时序稀疏奖励，直接 GRPO/策略优化容易整批失败、没有梯度信号。

GUI-R1: A Generalist R1-Style Vision-Language Action Model for GUI Agents

Offline RL
论文链接

动机

大多数 GUI agent 靠 SFT，需要大量数据，而且对未见界面泛化差。DeepSeek-R1 式 RLVR 启发作者用规则奖励提升 GUI action model。

方案

GUI-G2: Gaussian Reward Modeling for GUI Grounding

GroundingReward
论文链接

动机

传统坐标奖励把 GUI 元素当作 hit-or-miss 二值目标，导致奖励稀疏，并且“差 1 像素也失败”不符合真实点击行为。

方案

把目标 UI 元素建模为界面平面上的高斯分布。点奖励按距元素中心的指数衰减给分；coverage reward 衡量预测分布与目标区域重叠；方差根据元素尺寸自适应。

实验

在 ScreenSpot、ScreenSpot-v2、ScreenSpot-Pro 上超过 UI-TARS-72B，ScreenSpot-Pro 最大提升 24.7%。这类连续奖励成为后续 grounding RL 的常用思路。

Grounding
论文链接

BacktrackAgent: Error Detection and Backtracking Mechanism

Hybrid
论文链接

动机

现有 GUI agent 过度关注单步动作准确率，缺少发现错误、回退和恢复的机制，导致长任务中一次误点就失败。

方案

BacktrackAgent 加入 verifier、judger、reflector 模块：检测动作后状态是否偏离目标，判断是否需要回退，并反思生成修正动作。训练数据专门覆盖 action execution 后的 outcome pages，并加入 judgment reward。

实验

在 Mobile3M 和 Auto-UI 上提升任务成功率和 step accuracy。它代表“错误恢复作为可训练能力”而不是只做后处理。

VSC-RL: Variational Subgoal-Conditioned RL

Hybrid
论文链接

动机

视觉-语言 agent 的真实任务稀疏奖励、长时序依赖强，普通在线 RL 学习效率低。

方案

把任务重写为 variational goal-conditioned RL，优化 SubGoal Evidence Lower Bound。目标包括最大化 subgoal-conditioned return，以及约束与参考 goal-conditioned policy 的差异。VLM 自动把大目标分解成可行 subgoal。

实验

在多类 benchmark，包括移动设备控制和 Web 控制任务上超过 SOTA，并提升学习效率。综述将它列为移动/网页长任务的 subgoal-conditioned RL 代表。

UI-S1: Semi-online Reinforcement Learning

Hybrid
论文链接

动机

离线 RL 有稳定和便宜的优点，但缺 trajectory-level 真实反馈；在线 RL 有反馈但代价高、奖励稀疏。需要在静态轨迹上模拟在线偏移和修正。

方案

Semi-online RL 在离线 trajectory 中 rollout 当前模型输出，并用 Patch Module 修复模型动作偏离专家轨迹导致的 OOD 状态；奖励上同时用 discounted future return、step-level advantage 和 episode-level advantage；提出 SOP 作为 semi-online 性能代理指标。

Hi-Agent: Hierarchical Vision-Language Agents for Mobile Device Control

HybridMobile
论文链接

动机

移动端任务天然有层级结构：高层需要规划“先打开哪个 app、进入哪个页面、完成哪个子目标”，低层需要精确执行点击、输入和滑动。单一模型直接从指令到动作，容易在长任务中丢失全局目标，也难以同时兼顾语义推理和坐标/控件级执行。

方案

Hi-Agent 把移动 GUI agent 拆成高层 reasoning model 与低层 action model，并联合优化。高层负责分解任务、生成中间目标和上下文决策；低层根据当前屏幕和子目标执行具体 GUI 动作。训练上属于 hybrid/层级路线：用示范冷启动，再用 RL 强化高低层协同，缓解长时序 credit assignment。

实验

论文摘要报告 Hi-Agent 在 AITW 达 87.9%，AndroidWorld 达 21.0%，并在 AndroidControl-Low/High、AMEX、AndroidLab 等基准上展示跨任务泛化。综述把它作为“System-1 执行 + System-2 规划”式认知分层的代表。

Online RLMobile
论文链接

动机

WebAgent-R1: End-to-End Multi-Turn RL for Web Agents

Online RLWeb
综述引用为 EMNLP 2025，标题为 WebAgent-R1: Training Web Agents via End-to-End Multi-Turn Reinforcement Learning

动机

Web agent 的 reward 往往只在终局出现，单步优化容易学不到“先走看似无收益的步骤再完成任务”的 delayed gratification。

DART-GUI: Decoupled Training and Adaptive Data Curation

Online RLOS/Desktop
论文链接

动机

GUI RL 的主要瓶颈不是 GPU，而是多轮 GUI 环境 rollout 慢、有效交互数据少。

方案

DART 把训练系统解耦成四个异步模块：environment cluster、rollout service、data manager、trainer。引入 rollout-wise sampling、worker 级模型同步、按任务难度动态调整 rollout 数和长度、预收集困难任务成功轨迹、高熵 step 优先训练、truncated importance sampling 缓解 policy mismatch。

系统效率上 rollout GPU utilization 提升 1.6 倍、训练 throughput 提升 1.9 倍、环境利用率提升 5.5 倍。OSWorld 上 DART-GUI-7B 成功率 42.13%，比 base +14.61%，比开源 SOTA +7.34%。

Step-GUI Technical Report

DataReward
论文链接

动机

GUI 训练数据既要高质量又要低成本；同时实际部署需要跨设备标准接口并保护隐私。

方案

Environments

环境、容器与框架

2.1 Web / Browser 环境

常用环境：

  • MiniWoB++：早期 HTML5 synthetic micro-task sandbox，奖励稀疏但环境可并行，适合验证基础 RL。
  • WebShop：模拟电商，奖励可由商品属性重叠给出，适合语言 agent 与商品搜索。
  • Mind2Web / Multimodal-Mind2Web：真实网页快照与 DOM 标注，主要用于离线训练和 grounding/action candidate compression。
  • WebArena / VisualWebArena：自托管“模拟互联网”，包含购物、论坛、GitLab、地图等服务，能用后端脚本验证数据库/页面状态。
  • BrowserGym：把 WebArena、MiniWoB++、VisualWebArena、WebChoreArena 等统一成 gym-like API，提供 observation/action 标准化。

容器与框架：

  • WebArena/VisualWebArena 一般需要自托管服务，常见部署是 Docker/Docker Compose 起多个 Web service，再由 Playwright/浏览器自动化驱动页面。
  • BrowserGym/AgentLab 是 Python + Gymnasium 风格接口，底层常用 Playwright/Chromium，observation 包括目标、URL、DOM、accessibility tree、截图。
  • RL 框架侧常把 browser env 当 rollout worker，训练端用 GRPO/PPO/AWR/DPO 等；大规模时用异步队列隔离环境 I/O 和 GPU learner。

2.2 Desktop / OS 环境

常用环境：

  • OSWorld：真实 Ubuntu/桌面应用、多 App、多文件状态任务，369 个开放式桌面任务。
  • ScreenAgent / WindowsAgentArena / OfficeWorld：更偏桌面纯视觉、Windows 或 Office 场景。
  • TheAgentCompany、OmniACT、OSWorld-MCP、MCPWorld：企业任务、桌面+网页、API/GUI 混合评测。

容器与框架：

  • OSWorld 类环境常用 Docker + QEMU/虚拟机 + virtual framebuffer/VNC 截屏。Docker 负责隔离任务和依赖，QEMU/VM 提供真实 OS 环境，VNC/截图通道提供视觉 observation。
  • 控制动作常通过 PyAutoGUI、xdotool、VNC pointer/keyboard、Accessibility API 或系统 API。ComputerRL/UltraCUA/MAI-UI 开始把 GUI 原子动作和高层 API/MCP tool 合并成 hybrid action space。
  • 成功奖励不只看文本输出，而是检查文件、数据库、浏览器状态、应用侧 effect。

2.3 Mobile / Android 环境

常用环境：

  • AndroidEnv：标准 RL MDP for Android，连续触摸/手势，适合经典 RL，但时序长。
  • AndroidWorld：真实 Android App、116 个 programmatic tasks、动态参数化任务，用 ADB/SQLite/文件状态检查 reward。
  • AndroidLab、Mobile-Env、UISim、GUI Exploration Lab、MobileWorld：覆盖移动真实 App、模拟 UI 或导航图。

容器与框架：

  • 主体是 Android Emulator / AVD + ADB，而不是传统 Docker 容器。Docker 可以包外层服务，但真正的环境隔离来自 emulator snapshot、设备镜像、任务 setup/teardown。
  • 奖励常用 ADB 查询、SQLite 数据库检查、文件系统检查、Accessibility tree 或后台 evaluator。
  • 并行训练会启动几十到几百个 emulator，CPU/内存/I/O 是主要瓶颈；GPU 主要给 VLM inference/training。

2.4 世界模型 / Latent Rollout

代表：DynaWeb、WebSynthesis、WebWorld、Code2World、SimURA、UI simulator。

技术方案：

  • 学一个 action-conditioned dynamics model：输入当前 UI 表示和动作，预测下一网页/屏幕/DOM/可渲染代码。
  • 用 world model 做 MCTS、dream rollout 或 synthetic trajectory generation，再周期性回到真实环境校准。
  • Code2World 走 renderable code 路线，用 HTML/代码作为可控状态表示；WebWorld/DynaWeb 走网页状态生成路线；WebSynthesis 结合世界模型和 MCTS。

价值：

  • 绕过 live GUI 的 I/O wall。综述估算真实 web/desktop/mobile 单环境通常只有 0.2-2 steps/s，而 latent/world-model rollout 可到 10-100+ steps/s，仿真并行可达 10^3-10^4 steps/s 量级。
  • 主要风险是 sim-to-real gap，所以有效 pipeline 通常是：离线数据预训练 -> 世界模型/模拟环境训练 -> 少量真实在线校准。

2.5 RL 算法与训练系统

主流算法：

  • GRPO/RLVR：当前 GUI RL 最常见。优势是不用 critic，适合长视觉序列和大 VLM；代表 GUI-R1、UI-R1、InfiGUI-G1、GUI-Eyes、Mano。
  • DPO/Preference：适合从成功/失败轨迹对学习，训练稳定；代表 Agent Q、UI-TARS 部分阶段。
  • AWR/IQL/CQL/Q-learning：适合离线轨迹和 value reranking；代表 DigiRL、Digi-Q。
  • MCTS/Tree Search：适合轨迹合成和高质量数据搜索；代表 Agent Q、WebSynthesis。
  • Process/Progress Reward：解决长任务稀疏奖励；代表 ProgRM、Step-GUI、ClawGUI PRM、AgentPRM。

训练系统：

  • veRL、OpenRLHF、ReaL、AReaL、HybridFlow 是常见 RLHF/RLVR 系统底座。
  • vLLM/PagedAttention 用于 rollout inference；训练端常用 PyTorch FSDP、Megatron-LM 或 DeepSpeed；模型权重在 rollout actor 和 trainer 之间异步同步。
  • DART-GUI、ComputerRL、MAI-UI、AReaL 类系统都采用 actor-trainer 分离：CPU/浏览器/模拟器 worker 连续采样，GPU learner 从 replay buffer 消费轨迹。
  • 真正的扩展瓶颈是环境 I/O：截图、渲染、网络、DOM/状态解析、ADB/VM 文件 I/O，而不是单纯 GPU 数量。

Recipe

现在大家实际怎么做

一个比较标准的 pipeline 是：

  1. 数据冷启动：用 AitW、AndroidControl、Mind2Web、OmniACT、ScreenSpot、Ferret-UI、Explorer/AgentTrek/OS-Genesis 合成轨迹做 SFT 或 offline RFT。
  2. 统一动作空间：把 action 规范化为 click(x,y)、type(text)、scroll、swipe、back/home、finish，或进一步加入 API/MCP/tool calls。
  3. 奖励设计：低层 grounding 用坐标命中/IoU/高斯连续奖励；中层过程用 progress/process reward；终局用数据库、文件、URL、DOM、ADB 状态或 VLM-as-judge。
  4. 在线采样：Web 用 BrowserGym/WebArena + Playwright + Docker 服务；Desktop 用 OSWorld + Docker/QEMU/VNC；Mobile 用 Android emulator + ADB。
  5. 训练算法：小到中模型多用 GRPO/RLVR；需要偏好稳定时用 DPO；有离线轨迹时训练 value model 或用 AWR；长任务用 curriculum、replay、subgoal、PRM。
  6. 系统扩展：把环境 rollout 和模型训练异步解耦；大量环境 worker 并行；vLLM 做快速采样，FSDP/Megatron 做训练，队列或 replay buffer 管理轨迹。
  7. 真实部署：走 hybrid API-GUI，能调用 API 就调用 API，GUI 作为通用 fallback；加 verifier/backtracking/memory，避免一次错误导致整条任务失败。

Trends

关键趋势

  • 从“模仿人类点击”转向“可验证环境反馈上的自我改进”：GUI 比普通文本生成更适合 RLVR。
  • 从二值奖励转向连续、过程、多层奖励：GUI-G2/InfiGUI-G1/UI-AGILE/ProgRM/Step-GUI 都在解决 sparse reward。
  • 从单步 grounding 转向多轮任务和恢复：WebAgent-R1、MobileRL、DART-GUI、BacktrackAgent、Hi-Agent 是这条线。
  • 从 live rollout 转向 world model 和半在线：DynaWeb、WebWorld、Code2World、UI-S1 都是在绕 I/O wall。
  • 从纯 GUI 转向 API-GUI/MCP 混合：UltraCUA、ComputerRL、MAI-UI、ClawGUI 说明真实部署不会只靠鼠标键盘。
