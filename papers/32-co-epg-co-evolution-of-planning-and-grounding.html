Co-EPG: Co-Evolution of Planning and Grounding

Hybrid

TL;DR

Planner-Grounder 自迭代正反馈。Grounder 给 planner GRPO reward，planner 生成更好的多样数据反过来训练 grounder；更新后的 grounder 再给更准奖励，循环三轮。 在 Multimodal-Mind2Web 和 AndroidControl 上，三轮后超过现有 SOTA，不需要外部数据，并表现出每轮稳定自增强。

1. 要解决的问题

GUI agent 既要 planner 会制定策略，也要 grounder 会低层定位；现有方法常孤立优化两者，且 synthetic data 没被充分利用。

换成 GUI agent 的语言，这个问题通常落在三件事上：界面状态部分可观测、正确反馈稀疏且延迟、静态示范无法覆盖真实软件变化。论文的价值就在于把这些问题中的一个或多个转成可训练的闭环信号。

2. 任务形式：输入、动作、奖励

观察当前 GUI 视觉状态、任务指令和必要的历史上下文。
动作统一 GUI action schema，包括 click/type/scroll/swipe/back/finish 或工具调用。
奖励由终局完成、过程进展、动作合法性、坐标定位或 verifier 判断组成。

这个表是读 GUI Agent RL 论文最重要的入口：只要弄清楚模型看到了什么、能做什么、奖励从哪里来，就能判断它是算法贡献、环境贡献、奖励贡献还是系统贡献。

3. 方法与架构怎么跑

  1. Co-EPG 同时优化 planning 和 grounding：planner 产生高层步骤，grounder 把步骤落到具体 UI 元素。
  2. 两者 co-evolution 的意思是 planner 会根据 grounder 的实际可执行能力调整计划，grounder 也从 planner 的更好任务分解中学习。
  3. 如果只训练 planner，计划可能不可点；只训练 grounder，模型可能局部动作准但全局方向错。
  4. 这篇代表 GUI agent 中“语义规划”和“视觉落地”必须一起进化的方向。

因此，这篇论文不是简单地“让大模型点屏幕”，而是围绕 observation-action-reward 契约重写训练闭环：先让动作可执行，再让反馈可验证，最后让策略能从成功和失败中更新。

4. 训练、奖励与数据流

  • 混合路线先用 SFT/offline 数据冷启动，再用在线反馈、分层规划、API-GUI 混合动作或 verifier 做二次强化。
  • 它的核心取舍是：离线数据给稳定性，在线环境给真实反馈，系统结构负责把两者接起来。
  • 如果使用 GRPO/RLVR，模型会在同一状态采样多个候选动作，用相对奖励更新策略，避免训练单独 critic。

读实验时要特别注意 reward 口径：有的论文评估单步坐标，有的评估整条任务成功，有的只报告系统吞吐或数据合成质量。不同口径不能直接相加或横向混算。

5. 关键设计与创新点

  • 把 GUI 任务改造成可验证奖励问题，让策略优化不必依赖昂贵的逐步人工标注。
  • 通过层级、验证器、工具/API 或半在线机制，把长任务拆成更可控的子问题。

这些创新点的共同目标，是把 GUI 交互从一次性 prompt 调用变成可迭代优化的训练系统。

6. 实验怎么读

在 Multimodal-Mind2Web 和 AndroidControl 上，三轮后超过现有 SOTA，不需要外部数据，并表现出每轮稳定自增强。

这类实验通常需要同时看三组数字：第一是任务成功率或 grounding accuracy；第二是与 SFT、BC、prompting、GPT-4V/闭源模型或开源基线的对比；第三是环境成本，例如 rollout 速度、样本数量、标注成本和并行规模。

7. 复现或落地时要准备什么

  1. 先固定 action schema，并写一个能把模型输出解析成真实 GUI 操作的 adapter。
  2. 准备 verifier：至少要能判断终局成功；如果是 grounding 论文，还要能计算坐标/区域奖励。

如果只能先复现一小部分，建议优先复现 action schema、verifier 和一组可重置任务；没有这三件事，后续 RL 指标很难可信。

8. 局限与读法

  • 论文结果大多是作者自报或综述转述，跨 benchmark 的数值不可直接混算。

它属于系统融合路线：用层级结构、工具调用、验证器或半在线机制把真实 GUI 的长时序问题拆小。
